# Настройка Cursor для этого репозитория

Эта папка содержит проектные подсказки для Cursor-агентов и сотрудников, которые работают с `remap-api-specification` в Cursor.

## Контекст проекта

- `PROJECT.md` — основной входной файл для агентов. В нём описаны структура репозитория, CI-процессы, команды локальной проверки и важные соглашения проекта.
- Агентам стоит читать `PROJECT.md` перед нетривиальными изменениями, если нужен проектный контекст.

## Правила

Правила лежат в `.cursor/rules/` и описывают, как агентам следует работать в этом репозитории.

- `project-context.mdc` — говорит агентам начинать с `.cursor/PROJECT.md`, когда не хватает проектного контекста.
- `clarification-first.mdc` — требует задавать уточняющие вопросы, если требования неоднозначны или не хватает критичных данных.
- `quality-gates.mdc` — помогает провести финальные проверки перед отчётом о выполненной работе.
- `change-safety.mdc` — добавляет осторожность для изменений, чувствительных к совместимости.
- `security-baseline.mdc` — включает дополнительную внимательность к секретам, credentials, auth и security-sensitive конфигурации.
- `remap-project.mdc` — помогает держать документацию в актуальном состоянии при изменениях workflow, CI, тестов или структуры проекта.
- `plan-mode.mdc` — задаёт компактный формат планов реализации.
- `rule-governance.mdc` — описывает порядок изменения Cursor-правил.
- `dependency-governance.mdc` — описывает порядок изменения dependency manifest и lockfile.

## Skills

Skills лежат в `.cursor/skills/`.

### `add-entity`

Этот skill используется при добавлении новой API-сущности, ресурса или endpoint в OpenAPI-спецификацию, особенно если источником является файл `api-remap-1.2-doc/md/**/_*.md`.

Он помогает агентам:

- разбирать MD-документацию на схемы и endpoints;
- создавать entity, list, position, metadata, batch и delete schemas/paths;
- регистрировать новые paths и schemas в `src/openapi.yaml`;
- добавлять общие fixtures в `tests/fixtures/`, обновлять PHP/Java golden-тесты и PHP smoke-тесты;
- соблюдать текущие соглашения по `$ref` (`../../../openapi.yaml#/components/schemas/<SchemaName>` в request/response body path-файлов);
- запускать правильную последовательность Docker-проверок.

## `.cursorignore`

`.cursorignore` управляет тем, что Cursor не должен индексировать и использовать как широкий контекст кодовой базы. Он не удаляет файлы и напрямую не влияет на Git tracking.

В этом репозитории он исключает сгенерированные или тяжёлые локальные артефакты:

- зависимости (`node_modules/`, `tests/php/vendor/`);
- сгенерированные SDK (`clients/`);
- результат сборки OpenAPI-спеки (`dist/`, `build/`);
- build/test caches и локальные virtualenv;
- логи, env-файлы, IDE и OS noise.

### Нюанс со сгенерированным SDK

Так как `clients/` находится в ignore, Cursor обычно не будет использовать файлы сгенерированного SDK как индексируемый контекст. Это ускоряет повседневную работу со спецификацией и уменьшает шум, но важно помнить об этом, когда нужно проверить результат генерации SDK.

Если нужно понять, правильно ли сгенерировался SDK:

- сначала лучше смотреть generation logs, вывод `make generate-*`, golden tests, smoke tests и custom templates;
- для конкретного SDK-файла можно открыть его явно или попросить агента прочитать точный путь;
- для широкого ревью SDK можно временно убрать или сузить правило `clients/`, переиндексировать/открыть нужную папку SDK, а после ревью вернуть ignore-правило обратно.

Если SDK нужно ревьюить часто, можно использовать более узкий вариант:

```gitignore
clients/*/target/
clients/*/.openapi-generator/
clients/*/docs/
```

Так build/cache/docs останутся вне индекса, но Cursor сможет смотреть исходные файлы сгенерированного SDK.
