Ты — senior QA/SDK-аудитор для OpenAPI PHP генерации.
Нужно провести ПОЛНУЮ техническую и доменную проверку сгенерированного PHP SDK и найти системные дефекты, особенно:
- потерю union-типов (oneOf/anyOf),
- некорректную десериализацию массива/объекта,
- проблемы композиции схем с $ref,
- “ложно-валидные” пустые модели после deserialize.

## Контекст и источники истины

Используй как baseline:
1) Markdown-документация: {/remap-12-doc/md}
2) Маппинг разделов: {/remap-12-doc/config.json}
3) OpenAPI root: @src/openapi.yaml
4) Генерация SDK: @package.json
5) Локальный pipeline: @Makefile
6) Golden PHP tests: @tests/php/golden/SerializationTest.php

## Цель

Дать вердикт по пригодности PHP SDK к использованию и найти ВСЕ аналогичные ошибки по всему SDK, а не только в известных случаях.

## Обязательный план выполнения

### Шаг 1. Чистый воспроизводимый прогон
- Выполни lint/bundle/generate-php через локальный pipeline.
- Выполни golden и smoke тесты.
- Сохрани полный список ошибок/предупреждений.

### Шаг 2. Автопоиск риск-паттернов по ВСЕМУ SDK

Найди и проверь следующие паттерны глобально:

A) oneOf/anyOf в спецификации -> “схлопывание” в обычную object-модель в SDK
- Ищи в спецификации все oneOf/anyOf (schemas + requestBody + array items).
- Для каждого сопоставь с сгенерированными PHP model/API сигнатурами.
- Зафиксируй дефект, если union из спецификации исчез или заменился на одиночную модель без корректной полиморфии.

B) requestBody oneOf(object|array) -> сгенерирована только object-модель
- Найди все endpoints, где body допускает массив ИЛИ объект.
- Проверь generated request model и путь десериализации.
- Обязательно проверь, что payload-массив не превращается в “пустую валидную модель”.

C) anyOf-модели генерируются (AnyOf*) но не используются в реальном поле
- Найди все *AnyOf* классы.
- Проверь, есть ли поля родительских моделей, которые должны ссылаться на union-ветви, но типизированы “плоским” базовым классом.
- Проверь поведение во время выполнения через deserialize/get_class на показательной полезной нагрузке.

D) Недопустимая схема: $ref совмещен с другими свойствами
- Собери все места, где рядом с $ref есть соседние свойства (type/description/properties и т.п.).
- Классифицируй как антипаттерн спецификации и оцени риск нестабильной генерации между языками/версиями.

E) Error/errorOrArray-подобные схемы
- Найди все схемы, где union включает error object и array error object.
- Проверь сериализацию/десериализацию для обеих ветвей.
- Зафиксируй дефект, если верхнеуровневый массив не поддержан корректно.

### Шаг 3. Минимум проверок во время выполнения (обязательно)
Для каждого найденного дефекта:
- Укажи минимальный PHP-код с `ObjectSerializer::deserialize(...)` или вызовом SDK, который воспроизводит дефект.
- Укажи входные данные: JSON/payload и целевой класс/тип десериализации.
- Опиши ожидаемое поведение по документации.
- Опиши фактическое поведение SDK: полученный класс, данные в модели, ошибки валидации или их отсутствие.

### Шаг 4. Классификация причины
Каждую проблему отнеси к одной категории:
1) Ошибка/антипаттерн в OpenAPI spec
2) Ограничение openapi-generator (PHP)
3) Дефект customtemplates/php
4) Неверное допущение/хрупкость теста

Если неочевидно — пометь как “требует подтверждения”, но предложи самую вероятную первопричину с аргументами.

### Шаг 5. Итоговый вердикт и список задач
Сформируй:
- Вердикт: “SDK usable / conditionally usable / blocked”
- Что уже нормально для использования
- Что блокирует использование (High)
- Приоритезированный список исправлений (P0/P1/P2), где для каждого:
  - минимальный фикс,
  - где править (spec/generator/template/tests),
  - как проверить, что исправлено (конкретный тест/скрипт).

## Формат результата (строго)

1) Базовые проверки (версии, команды, окружение)
2) Таблица/список найденных проблем:
   - Серьезность
   - Симптом
   - Где найдено (путь в спецификации + сгенерированный файл)
   - Как воспроизвести
   - Категория первопричины
3) Отчет о покрытии:
   - сколько oneOf/anyOf конструкций проверено,
   - сколько requestBody union-кейсов проверено,
   - сколько $ref-sibling антипаттернов найдено
4) Финальный вердикт
5) Топ-10 конкретных действий (в порядке приоритета)

## Критерии готовности

- Прогон полностью воспроизводим командами.
- Найдены не только известные кейсы, но и аналогичные паттерны по всему SDK.
- Для каждой проблемы высокой/средней серьезности есть доказательство (спецификация + сгенерированный код + проверка во время выполнения/тест).
- Есть четкий вердикт и исполнимый список задач.
